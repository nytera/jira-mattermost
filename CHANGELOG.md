# Changelog

Все значимые изменения проекта документируются в этом файле.

Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/),
проект придерживается [семантического версионирования](https://semver.org/lang/ru/).

Этот раздел охватывает все изменения, сделанные поверх исходного импорта
(«оригинального мастера» — коммит `5b55e83 Initial Mattermost Jira incident bot`,
он же текущий `origin/main`).

## [Unreleased]

### Добавлено

**Реакции-эмодзи**
- Реакция `man_gesturing_no` → проставляет Jira-поле «Валидность» = `Ложный`,
  `arrows_counterclockwise` → `Ожидаемый` (имена настраиваются через
  `MATTERMOST_FALSE_INCIDENT_REACTION_NAME` /
  `MATTERMOST_EXPECTED_INCIDENT_REACTION_NAME`). Лёгкий путь: пишет только поле
  «Валидность» и отвечает в тред, без поста в инцидент-канал. Побеждает
  последняя реакция; повторная такая же реакция игнорируется (нет дублей).
- Чекмарк-реакции (`white_check_mark` / `heavy_check_mark` /
  `ballot_box_with_check`) на корне треда в инцидент-канале → проставляют
  end-time в Jira и (при настроенном LLM) генерируют постмортем; на корне
  ручного треда — создают Jira-задачу. Чекмарк на ответе в треде или вне
  инцидент-канала игнорируется.

**Интерактивные кнопки и меню в треде** (при заданном `SERVICE_PUBLIC_URL`)
- Карточка алерта: меню «Выбрать валидность ▼» (`Ложный`/`Ожидаемый`/`Валидный`),
  кнопки `🚨 Инцидент` и `📝 Summary`, отдельный блок `💬 Обратная связь по алерту`.
- Карточка инцидента (инцидент-канал): для ручных инцидентов — `➕ Создать задачу`,
  затем меню валидности, `🏁 Завершить`, `📝 Саммари`.
- Confirm-диалоги Mattermost перед `🚨 Инцидент` и `🏁 Завершить`.
- Смена надписей по состоянию: `🚨 Инцидент` → `✅ Подтверждён`,
  `🏁 Завершить` → `✅ Завершено`.
- Форма обратной связи (Mattermost dialog с textarea): сохраняется в БД, бот
  пишет в тред «Получили обратную связь от <имя>». Доступна всем без авторизации.
- Кнопка `📝 Summary` / `Саммари`: отправляет тред в LLM и публикует краткое
  саммари ответом в тред (без LLM — эфемерный no-op).
- Тумблер `INTERACTIVE_BUTTONS_ENABLED`: при `false` карточки не прикрепляются
  даже с заданным `SERVICE_PUBLIC_URL` — режим «только эмодзи».
- Стабильные id действий в `context`, чтобы переименование кнопки не ломало уже
  опубликованный пост.

**Ручные инциденты в инцидент-канале**
- Корневой пост реального пользователя (не бот/вебхук) получает карточку
  `➕ Создать задачу`; Jira-задача создаётся по клику, карточка заменяется на
  полный набор контролов. Идемпотентно.
- Пинг дежурных: `MATTERMOST_DUTY_MENTION` ставится в текст сообщения над
  карточкой (именно в text, чтобы `@group`-пинг сработал).

**Авторизация**
- `MATTERMOST_AUTHORIZED_USERNAMES` — allowlist логинов: бот реагирует на
  реакции/нажатия/slash только от перечисленных пользователей (логины
  резолвятся в user-id на старте). Пусто = разрешено всем. Обратная связь
  открыта всем. При полном провале резолва гейт отключается (fail-open).

**LLM: постмортемы и саммари**
- Клиент к OpenAI-совместимому API (`LLM_BASE_URL`, `LLM_API_TOKEN` /
  `CORELLM_API_TOKEN` / `OPENAI_API_KEY`, `LLM_MODEL`, `LLM_MAX_TOKENS`,
  `LLM_THREAD_MAX_CHARS`, `LLM_STREAM`, `LLM_READ_TIMEOUT`).
- Генерация постмортема по всему треду инцидента: PM-шаблон в описании задачи,
  полный отчёт — комментарием в Jira, краткое резюме — ответом в тред.
- Генерация краткого саммари треда по запросу.
- Стриминг SSE с авто-fallback на обычный chat-completion, если прокси не
  поддерживает поток.

**Jira (on-prem 9.x Data Center, REST v2)**
- Резолв полей по человекочитаемому имени (в т.ч. русскому, напр. «Валидность»)
  в `customfield_*` через `GET /rest/api/2/field` с кэшем; готовый
  `customfield_NNNNN` используется как есть.
- Резолв допустимых значений `select`/`radiobuttons` через issue-type
  createmeta (в т.ч. постраничный), сопоставление по value/name без учёта
  регистра; отсутствующая опция → понятная нертраемая ошибка.
- `JIRA_START_FIELD` — date-time поле времени прихода алерта; `JIRA_END_FIELD` —
  время реакции/закрытия. Формат ISO 8601 с offset `+0300` и дробными секундами.
- `JIRA_SOURCE_FIELD` (`Crit alert`) и `JIRA_IS_CRIT_ALERT_FIELD` (`Да`)
  проставляются только для алерт-задач.
- `JIRA_CONFIRMED_STATUS_ID` — опциональный transition в статус подтверждённого
  инцидента.
- Описание задачи на русском в wiki-markup (заголовок, таблица параметров,
  ссылка на исходное сообщение); отдельный PM-шаблон описания для постмортемов.
- Тестовый режим `JIRA_CREATE_ENABLED=false`: задача не создаётся, в БД пишется
  stub-ключ (`JIRA_STUB_ISSUE_KEY` либо сгенерированный `PROJECT-NNNNN`); все
  изменяющие операции с задачей — no-op.
- Стартовый preflight: проверка резолва всех полей, issue-type и option-значений.

**Сообщение инцидента**
- Полностью на русском: заголовок `##### 🔴 Инцидент открыт` (при завершении —
  `##### 🟢 Инцидент закрыт`, заголовок редактируется на месте), серый блок с
  деталями (задача Jira, исходный алерт, кто подтвердил — живой `@mention`,
  время), пересылка вложений исходного алерта отдельным блоком.

**Операции и инфраструктура**
- Debug-admin (`DEBUG_ADMIN_ENABLED`): SPA `GET /debug/admin` с карточками-
  метриками, таблицей алертов, поиском/фильтрами, авто-обновлением, модалкой
  деталей, вкладкой логов и созданием задачи из ссылки; JSON API (summary,
  alerts, детали, recreate, create-from-link, logs).
- Эндпоинты `POST /mattermost/actions/alert` (кнопки/меню) и
  `POST /mattermost/dialogs/feedback` (форма обратной связи).
- Стартовый preflight (проверки БД / Mattermost / Jira / LLM) и pending-work
  loop, добивающий упавшие создания задач и отложенные подтверждения.
- Логирование: `LOG_FORMAT` (`json`|`text`) с компактным `TextFormatter`,
  `EventLogger`/`get_logger`, in-memory `LogRingBuffer` для вкладки логов.
- Таблица `alert_feedback`, новые колонки `alert_tickets`
  (`mattermost_alert_title`, `validity_label`, расширенные state-машины),
  онлайн-`ALTER TABLE` на старте для существующих БД.
- `INCIDENT_TIMEZONE` (по умолчанию `Europe/Moscow`) — единая backend-таймзона
  для всех сохраняемых/отображаемых времён.
- Инструментарий: `ruff` (lint+format) и `pytest-cov` в extra `[test]`,
  прод-вариант `docker-compose_prod.yml`, healthcheck/restart-policy в compose.

**Уведомления в треде (последнее)**
- Все текстовые уведомления бота в треде теперь рендерятся как attachment-боксы
  с цветной полосой (`NOTICE_ATTACHMENT_COLOR`), а не голым сообщением; текст
  дублируется в `fallback` для push-уведомлений и превью. Боксинг пропускается
  для постов с интерактивными карточками и для сообщений с `@mention` (чтобы
  пинг сработал).

### Изменено

- **Jira REST API v3 → v2** и **ADF → wiki-markup** во всех описаниях и
  комментариях.
- **Авторизация Jira** переведена на Bearer Personal Access Token (вместо
  Basic-auth парой email + token).
- Ребрендинг пользовательских строк: «Mattermost» → «Band» (описания, комментарии,
  ответы в треде, slash-ответы); кодовые идентификаторы и `MATTERMOST_*`-переменные
  не тронуты.
- Сообщение инцидента переформатировано (русский заголовок + серый блок деталей +
  блок вложений) вместо прежнего английского `### Confirmed incident` одним текстом.
- Уведомление о валидности больше не повторяет ссылку на задачу — теперь просто
  «Поле «Валидность» обновлено: <значение>.» (ссылка уже есть в верхнем сообщении
  «Создана задача» того же треда).
- Поведение по умолчанию: бэкфилл при старте выключен
  (`ENABLE_BACKFILL_ON_STARTUP=false`), `BACKFILL_RECENT_POSTS_LIMIT` 100 → 0 —
  бот обрабатывает только новые события.
- `JIRA_VALID_INCIDENT_FIELD` принимает имя поля или `customfield`-id (читается
  из `JIRA_VALID_INCIDENT_FIELD` / `..._NAME` / `..._ID`); пример в `.env.example`
  сменён на `Валидность`.
- Извлечение заголовка алерта: чистка ведущих эмодзи/символов, разбор
  Grafana-ссылок.
- Рефакторинг инфраструктуры (поведение сохранено): общий базовый клиент
  `AsyncApiClient` с централизованными ретраями/логированием для Mattermost и
  Jira; единый `EventLogger`; мутаторы репозитория через общий `_mutate`;
  выделены модули `jira_payload.py`, `debug_admin.py`, `http.py`, `llm.py`,
  `summary.py`, `postmortem.py`.
- `docker-compose.yml` (dev): сервисы переименованы в `*_dev`, порт бота
  `8085:8080`, проброс порта Postgres убран.

### Исправлено

- Авторизация делает **fail-open**, если ни один из заданных логинов не
  резолвится (например, все с опечаткой) — раньше это блокировало всех.
- Чекмарк в алерт-канале больше не уходит в алерт-ветку и не создаёт по ошибке
  Jira-задачу.
- Явно выбранная валидность `Ложный`/`Ожидаемый` сохраняется при
  завершении/постмортеме (валидность и подтверждение — независимые оси).
- Заголовок инцидента становится зелёным, даже если генерация постмортема упала,
  но end-time в Jira уже проставлен.
- Тестовый режим: изменяющие Jira-операции стали no-op — раньше они получали 404
  по несуществующему stub-ключу и обрывали `confirm_incident` после поста в
  инцидент-канал, но до ответа в треде.
- Разбор option-полей в `get_valid_incident` (словарь `{"value": ...}` вместо
  ожидания булева значения).
- Пагинация createmeta безопасно завершается (нет бесконечного цикла и потери
  полей со второй страницы).
- LLM-стриминг устойчив к прокси без поддержки SSE (fallback на JSON) и к
  пустому ответу (явная ошибка вместо «тихого» пустого постмортема).
- Транспортные ошибки httpx (таймауты/обрывы) оборачиваются в retryable
  `ApiError` — websocket-loop переживает сетевые сбои.
- Гонки вставки (`create_or_get_alert` / `create_or_get_incident_thread`)
  обрабатываются через `IntegrityError` + rollback + повторную выборку.
- Summary: полный год (`%Y`), берётся первая непустая строка, срезается ведущий
  статус-эмодзи (`🔴`).
- Resolved-алерты (первая строка с `✅`) не создают Jira-задачу.

### Удалено

- Переменная `JIRA_EMAIL` (переход на Bearer PAT).
- ADF-построители описаний/комментариев (`adf_document`, `_paragraph`,
  `_text_node`, `_link_paragraph`).
- Старый английский шаблон сообщения инцидента `### Confirmed incident`.
- Мёртвый код: `domain.utc_now`, `mattermost_ms_from_datetime`,
  `tickets_to_ids`, неиспользуемые Jira-фоллбэки совместимости.
